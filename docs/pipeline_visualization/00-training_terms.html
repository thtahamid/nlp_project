// what counts as one step

Micro-batch → Step → Optimizer update

With batch_size=1 and grad_accum=2, the trainer processes 2 samples (forward+backward) before one optimizer step. That's why you can keep batch=1 to fit VRAM but still get the gradient quality of batch=2.

Why batch_size=1 in our setup? Each Qwen2.5-VL sample contains a base64-decoded handwriting image and ~1k tokens. Two samples already nudge a 23 GB A10G to its limit. Gradient accumulation lets us simulate batch=2 (or higher) without OOM.

Train loss vs. eval loss — Run 02

Loss = how wrong the model's predictions are (lower = better). Two curves matter: train (what the model sees) and eval (what it doesn't). Click a phase to learn what to look for.

Warm-up (steps 0–280, epoch 0–0.5). Train loss collapses from 8.01 → 1.56 (≈80%) as the LoRA adapter learns the most obvious patterns. Learning rate is ramping up linearly; eval is not measured yet.
// evaluation — what it really is

Evaluation = "test on data the model has never seen"

During training you also have a held-out eval set (here: 280 samples in data/eval/eval.jsonl). Once per epoch the trainer freezes weights, runs the model over the eval set, and reports the mean loss. This is the only honest signal for "is the model getting better at things it hasn't memorized?"

Two senses of "evaluation" in this project.
① During training — notebook 05 reports eval_loss per epoch (raw cross-entropy on next-token prediction).
② Post-training — notebook 06 runs the model end-to-end and computes CER, WER, exact-match, valid-JSON rates. These are task metrics; loss is a proxy.
// checkpoints — saved snapshots

checkpoint-1120 vs. checkpoint-1680

A checkpoint = the full set of trainable weights at a specific step, dumped to disk so you can restore (or evaluate) that exact model state later. Run-02 saves one per epoch.

// over-fit, under-fit, just right

How to read the train/eval gap

Underfit
both losses high
Train ≈ eval, both stuck high. Model hasn't learned enough. Fix: train longer, raise LR, raise LoRA rank, add layers to target_modules.
Just right
small, stable gap
Both decreasing together; eval still trending down. Run-02 epoch 2 is here.
Mild overfit
train ↓, eval flattens
Eval starts to plateau or nudge up while train keeps falling. Run-02 epoch 3 is here (train 1.46, eval 1.64 — gap widened to 0.18). Stop earlier or add regularization.
Severe overfit
eval climbs sharply
Train near zero, eval rising fast. Model memorized the training set. Fix: early stopping, more data, dropout up, weight decay, lower LR.
// all the terms · alphabetical
batch_size
samples per micro-batch
How many samples the model sees in one forward+backward pass. Bigger = smoother gradients, more VRAM.
Run-02: 1 (image-heavy samples; A10G can't fit more)
grad_accum
gradient_accumulation_steps
Sum gradients over N micro-batches before applying. Simulates a bigger batch without the memory cost.
Run-02: 2 → effective batch = 2
effective_batch
batch_size × grad_accum
What actually shows up in one optimizer step. This is the number you compare against papers' "batch size".
Run-02: 1 × 2 = 2
step
one optimizer.step()
One actual update to the model weights. The trainer's progress bar counts these — not samples.
Run-02: 1,680 total
epoch
one pass over train set
All training samples seen exactly once. With effective batch=2 and 1,120 samples, one epoch = 560 steps.
Run-02: 3 epochs × 560 = 1,680 steps
learning_rate
step size per update
How aggressively each gradient nudges the weights. Too high = unstable; too low = slow / stuck.
Run-02: peak 2e-5
lr_scheduler
how the LR changes over time
cosine = warm up linearly, then decay along a cosine curve to ≈0. Smoother than constant LR.
Run-02: cosine + 10% warmup
warmup_ratio
fraction of steps spent ramping LR
First X% of training, LR rises from 0 → peak. Prevents early instability when gradients are wild.
Run-02: 0.1 (first 168 steps)
train_loss
loss on training batches
Mean loss over the most recent logged steps. Falling = the model is fitting the data it sees.
Run-02: 8.01 → 1.46
eval_loss
loss on held-out eval set
Mean loss on data the model has never trained on. The honest signal of generalization.
Run-02: 1.6350 → 1.6248 → 1.6387
final eval loss
eval at the last step
Reported at end of training. Not always the best — if overfitting started, an earlier checkpoint may generalize better.
Run-02: 1.6387 (worse than epoch 2)
grad_norm
L2 norm of gradients
Magnitude of the gradient vector pre-clipping. Watch for spikes (instability) or sudden drops (vanishing grads).
Run-02: drifted 2 → 4 across the run
max_grad_norm
gradient clipping threshold
If grad_norm exceeds this, the gradient is rescaled down. Prevents exploding-gradient blow-ups.
Run-02: 1.0
checkpoint
snapshot of weights
Full save of trainable params at a step. Restore for resume, evaluate, or ship.
Run-02: ckpt-1120, ckpt-1680
best checkpoint
lowest eval_loss across the run
The state of the model when it generalized best. Often not the last one.
Run-02: checkpoint-1120
save_strategy
when to dump checkpoints
epoch = save at end of each epoch. steps = save every N steps.
Run-02: epoch (limit=2 keeps disk small)
eval_strategy
when to run evaluation
epoch = eval at end of each epoch. no = never (skip eval entirely).
Run-02: epoch · 280 samples in ~37 s
logging_steps
how often the trainer prints
Train loss + grad norm + LR are logged every N steps. Smaller = more granular curves but more I/O.
Run-02: 10
gradient_checkpointing
memory-time trade-off
Discards activations during forward pass and re-computes them on backward. Cuts VRAM ~50% at ~25% speed cost.
Run-02: on (required for 7B on A10G)
bf16
brain-float 16-bit
Half-precision with the same exponent range as fp32. Faster matmuls; safer than fp16 for fine-tuning.
Run-02: compute dtype = bf16
weight_decay
L2-style regularization
Shrinks weights slightly each step. Discourages overfit. Often skipped for small LoRA adapters.
Run-02: 0.0 (LoRA already tiny)
CER · WER
character / word error rate
Edit-distance / reference-length. Range [0, 1] when computed with jiwer. Lower is better.
Used in: notebook 06 (post-training)
exact_match
prediction == reference
Strictest metric. 100% means letter-perfect transcription.
Run-01 eval: 0% (model still drifts)
seed
RNG seed
Fixes data shuffle, dropout masks, init noise — so the same command gives the same numbers.
Run-02: 42
