Micro-batch → Accumulate → Optimizer update

Why split a batch at all? Big batches give smoother gradients but eat VRAM. If a batch of 16 won't fit, run batch_size=4 with grad_accum=4 — same effective batch, fits the card. The model can't tell the difference.
// reading a loss curve

Train loss vs. eval loss — the canonical shape

Loss = how wrong the model's predictions are (lower = better). Two curves matter: train (what the model sees and learns from) and eval (what it never trains on — the honest signal). The chart below is the textbook shape for an LLM fine-tune: a steep warm-up, a fitting plateau, then a gentle overfit. Click a phase to learn what to watch for.

Warm-up. The first chunk of training: train loss collapses fast as the model picks up the most obvious patterns. Learning rate is ramping up linearly from zero. Eval isn't usually measured yet — too noisy this early. What to watch: the curve should drop steeply, not stall.
// evaluation — what it really is

Evaluation = "test on data the model has never seen"

Before training, you set aside a slice of the dataset and never touch it during fitting. Once per epoch the trainer freezes the weights, runs the model over that held-out eval set, and reports the mean loss. It's the only honest signal for "is the model getting better at things it hasn't memorized?"

Two senses of "evaluation" you'll see in any project.
① During training — a fast eval_loss reported per epoch (raw cross-entropy on next-token prediction). Cheap, runs on the same loop as training.
② Post-training — a separate evaluation pipeline runs the model end-to-end and computes task metrics: CER, WER, exact-match, BLEU, accuracy — whatever fits the task. These measure what users actually care about; loss is a proxy.
// checkpoints — saved snapshots

Best checkpoint vs. final checkpoint

A checkpoint = the full set of trainable weights at a specific step, dumped to disk so you can restore (or evaluate) that exact model state later. Trainers typically save one per epoch — and the numbered subfolder name (e.g. checkpoint-1120) is the global step count at save time.

Best checkpoint
lowest eval loss across the run
The state where the model generalized best. This is the one you ship. Often falls in the middle of training, not the end.
Final checkpoint
last save, after training ends
The most-trained version. Sometimes also the best — but if eval loss climbed in the last epoch (mild overfit), an earlier checkpoint will do better on real data.
Rule of thumb: always evaluate at least two checkpoints — the best (by eval loss) and the final — and compare their task metrics. Don't trust "more training = better."
// over-fit, under-fit, just right

How to read the train/eval gap

Comparing the two curves is the single most useful diagnostic in training. The shape of the gap tells you what to do next.

Underfit
both losses high
Train ≈ eval, both stuck high. The model hasn't learned enough. Fix: train longer, raise LR, raise model capacity (e.g. higher LoRA rank), add more layers to target_modules.
Just right
small, stable gap
Both losses are decreasing together. Train is a little lower than eval, but eval is still trending down. Keep training — you haven't hit the sweet spot yet.
Mild overfit
train ↓, eval flattens
Eval starts to plateau or nudge up while train keeps falling. The gap widens. Stop here — earlier checkpoints are your best bet. Or add a touch of regularization (dropout, weight decay).
Severe overfit
eval climbs sharply
Train near zero, eval rising fast. The model has memorized the training set. Fix: early stopping, more data, more dropout, more weight decay, lower LR — and don't ship the final checkpoint.
// all the terms · alphabetical
batch_size
samples per micro-batch
How many samples the model sees in one forward+backward pass. Bigger = smoother gradients, more VRAM.
Typical: 1–8 for 7B+ LLMs on consumer GPUs
grad_accum
gradient_accumulation_steps
Sum gradients over G micro-batches before applying. Simulates a bigger batch without the memory cost. Lets you decouple "what fits in VRAM" from "what makes a good gradient."
Example: B=2, G=4 ⇒ effective batch 8
effective_batch
batch_size × grad_accum
What actually shows up in one optimizer step. This is the number you compare against the literature's "batch size".
Formula: effective_batch = B × G
step
one optimizer.step()
One actual update to the model weights. The trainer's progress bar counts these — not samples.
≠ sample, ≠ micro-batch
epoch
one pass over the train set
Every training sample seen exactly once. The number of epochs is usually small for fine-tuning (1–5) and larger for from-scratch training (tens to hundreds).
Formula: steps/epoch = ⌈ N / (B·G) ⌉
learning_rate
step size per update
How aggressively each gradient nudges the weights. Too high = unstable; too low = slow / stuck.
Typical fine-tune: 1e-5 – 5e-5
lr_scheduler
how the LR changes over time
cosine = warm up linearly, then decay along a cosine curve to ≈0. Smoother than constant LR. Other choices: linear, constant, polynomial.
Default: cosine + small warmup
warmup_ratio
fraction of steps spent ramping LR
First X% of training, LR rises from 0 → peak. Prevents early instability when gradients are wild and the model is mostly random.
Typical: 0.03 – 0.1
train_loss
loss on training batches
Mean loss over recently-logged steps. Falling = the model is fitting the data it sees. Falling fast then plateauing is normal.
What to watch: downward, then flat
eval_loss
loss on held-out eval set
Mean loss on data the model has never trained on. The honest signal of generalization. Compare against train_loss to spot overfit.
What to watch: reaches a minimum, then rises
final eval loss
eval at the last step
Reported at end of training. Not always the best — if overfitting kicked in, an earlier checkpoint will generalize better. Always compare to the minimum.
grad_norm
L2 norm of gradients
Magnitude of the gradient vector pre-clipping. Big spikes = instability; sudden drops to ~0 = vanishing gradients.
Healthy: stable, in single digits
max_grad_norm
gradient clipping threshold
If grad_norm exceeds this, the gradient is rescaled down. Cheap insurance against exploding-gradient blow-ups.
Default: 1.0
checkpoint
snapshot of weights
Full save of trainable params at a step. Restore for resume, evaluate, or ship.
Folder name: checkpoint-{step}
best checkpoint
lowest eval_loss across the run
The state of the model when it generalized best. Often not the last one — and that's a feature, not a bug.
save_strategy
when to dump checkpoints
epoch = save at end of each epoch. steps = save every N steps. Pair with save_total_limit to keep disk usage in check.
eval_strategy
when to run evaluation
epoch = eval at end of each epoch. steps = every N steps. no = never (skip — fastest, but you fly blind).
logging_steps
how often the trainer prints
Train loss + grad norm + LR are logged every N steps. Smaller = more granular curves but more I/O. 10 is a common balance.
gradient_checkpointing
memory-time trade-off
Discards activations during forward pass and re-computes them on backward. Cuts VRAM ~50% at ~25% speed cost. Practically required for 7B+ models on consumer GPUs.
bf16 / fp16
half-precision compute
16-bit floats — faster matmuls, half the activation memory. bf16 has fp32's exponent range and is safer for fine-tuning; fp16 is faster on older GPUs but can overflow.
weight_decay
L2-style regularization
Shrinks weights slightly each step. Discourages overfit. Often skipped for small LoRA adapters; kept (~0.01) for full fine-tuning.
dropout
stochastic regularization
Randomly zeros a fraction of activations during training so the model can't lean on any single neuron. A small dropout in LoRA (~0.05) goes a long way.
CER · WER
character / word error rate
(Substitutions + Deletions + Insertions) / N_reference. Lower is better, 0 is perfect. Can exceed 1.0 when the hypothesis is much longer than the reference — insertions simply outnumber reference tokens. Used post-training, never as the optimization target.
Pitfall: a short 5-char ref vs. an 85-char hallucination ⇒ CER ≈ 16
exact_match
prediction == reference
Strictest metric. 100% = letter-perfect output. Useful when the task has one correct answer (transcription, structured extraction).
seed
RNG seed
Fixes data shuffle, dropout masks, init noise — so the same command gives the same numbers. Essential for fair experiments.
Convention: 42 (the obvious one)
stop_strings
early-termination trigger at inference
A list of substrings that halt model.generate() the moment any of them appears in the decoded stream. Cheapest way to cap over-production when you know the valid output shape — e.g. stopping at "} when the task returns JSON. Saves wall-clock and stops trailing hallucinations from tanking CER/WER.
Usage: generate(..., stop_strings=['"}'], tokenizer=tok)
JSON-prefix forcing
prompt-side structural scaffolding
Append the opening of the expected JSON (e.g. {"transcription":") to the prompt before generation. The model then only has to complete the value and close the braces, which turns a weak "emit JSON" instruction into a near- guaranteed parseable output. Pairs naturally with stop_strings=['"}'].
Effect: valid_json_rate → 100% without any format fine-tuning
LM prior / fallback
pretraining leak when conditioning is weak
When the task-specific signal (e.g. a handwritten image) is ambiguous or low-information, the decoder's next-token distribution is dominated by the base model's pretraining prior and it emits a high-probability generic sentence instead of the true answer. Looks like hallucination but is actually the LM doing its default job.
Symptom: the same canned phrase repeats across many failed samples, and grep-ing the phrase against train.jsonl returns zero matches
length bias
model inherits ref-length distribution
If training targets cluster around, say, 60 characters, the model learns an implicit "a transcription should be ~60 chars" prior. On short-reference samples it keeps generating past the true answer because stopping earlier has low probability under that prior. Fix at the data layer (balance short/long examples) or at inference (stop_strings, length penalty).
Watch: len(ref) histogram on train vs. eval
repetition_penalty
penalty on already-emitted tokens
Multiplier > 1.0 that down-weights the logits of tokens that have already appeared in the sequence. Cheap guard against degenerate loops ("…مثل الحوار مثل الحوار مثل…"). Too high (> 1.3) makes the output feel unnatural — 1.05–1.15 is the usual sweet spot.
Typical: 1.1 with greedy decoding
no_repeat_ngram_size
hard block on repeated n-grams
Forbids any n-gram of length N from appearing twice in the same generation. Stronger than repetition_penalty — useful when a model has a specific phrase it likes to loop on. Complements rather than replaces EOS-based stopping.
Usage: no_repeat_ngram_size=6 blocks any 6-token loop
KV cache
reuse attention keys/values at inference
At each generation step, the transformer caches the keys and values for all previous tokens so it only computes attention for the new token. Single biggest per-token speedup at inference. Training with gradient_checkpointing=True disables use_cache — remember to flip it back on before eval: model.config.use_cache = True.
Effect: O(N²) → O(N) per-token, often 5–10× faster
