Δ what changed · the fine print
2-VL → 2.5-VL · vision overhaul Window attention (8×8) replaces full attention in most ViT layers (only 4 full-attention layers retained) → throughput up, VRAM down on high-res inputs. LayerNorm → RMSNorm, GELU → SwiGLU so the ViT and LLM share primitives (FSDP / kernel fusion friendly).
2-VL → 2.5-VL · temporal + spatial M-RoPE temporal IDs switch from frame-index to absolute wall-clock timestamps, enabling dynamic FPS sampling and second-level event localisation. Spatial coords (bboxes, points) now in real image pixels — no normalisation — enabling structured JSON detection output.
2.5-VL → 3-VL · position Interleaved-MRoPE: all three axes (T/H/W) get full-frequency allocation instead of disjoint bands — fixes long-video positional blur. Video timestamps represented as text tokens interleaved with frames, replacing T-RoPE's implicit encoding.
2.5-VL → 3-VL · scale Context jumps from 32 K → 256 K native, extendable to ~1 M via YaRN. MoE tiers (30B-A3B, 235B-A22B) introduced — first vision MoE. Each tier ships an -Instruct and a reasoning-tuned -Thinking variant. OCR coverage expanded from 19 → 32 languages.
trap · processor signatures are not interchangeable qwen_vl_utils.process_vision_info evolved across generations. Qwen2.5-VL adds return_video_kwargs=True for dynamic-FPS alignment; Qwen3-VL adds timestamp-token bookkeeping. Always use the model's own AutoProcessor — never reuse a processor from a different generation.

2.5 → 3 · attention + MoE rework QK-Norm added to attention for training stability. QKV bias removed (2.5 had bias). Unified thinking/non-thinking in a single checkpoint — toggle via enable_thinking flag or /think / /nothink soft tokens. MoE drops shared experts and adds global-batch load-balancing loss. Pretraining scales to ~36 T tokens across 119 languages.
3 → 3.5 · two paradigm shifts at once (1) Natively multimodal by default — every Qwen3.5 checkpoint is image-text-to-text; the "text-only vs VL" split from earlier generations is gone. (2) Attention moves to a Gated-DeltaNet + softmax-attention hybrid, pattern: 3 × (DeltaNet + FFN) → 1 × (GatedAttn + FFN), repeated. This is linear-attention territory → 256 K context becomes tractable on commodity hardware.
3.5 → 3.6 · agentic tuning Architecture lineage preserved. Main differences are training-data composition (agentic coding trajectories, tool-use traces) and post-training RL on million-agent environments. Qwen3.6-35B-A3B is the open-weight flagship so far; full arch specs [UNVERIFIED · arXiv tech report not yet out].
trap · Qwen3.5 QLoRA is broken Unsloth explicitly warns that 4-bit QLoRA is unstable on Qwen3.5 because the Gated-DeltaNet hybrid layers don't round-trip cleanly through bnb 4-bit. Use bf16 LoRA or 8-bit. Hard constraint on 8 GB GPUs: you can't 4-bit-squeeze Qwen3.5-4B onto an RTX 4060.
our project's 9B scaffold Notebooks 08/09 (notebook/08_finetuning_qwen3.5-9b.ipynb) target Qwen/Qwen3.5-9B — that's a real HF release, 10 B params, BF16 weights ~19.3 GB, 6 M+ monthly downloads. Adapter pipeline identical to the 2.5-VL track; what differs is dep versions (needs transformers ≥ 4.57), trust_remote_code=True on older stacks, and the QLoRA caveat above.
03 native multimodality · the omni family

Projector bottleneck → shared tokenizer → Thinker-Talker

"Native multimodality" is not just better finetune data. It means the LLM backbone sees <text, image, audio, video> interleaved tokens from pretraining, in a shared tokenizer with first-class audio/vision special tokens, under a unified positional embedding (TMRoPE) that aligns time across modalities.

Why this matters for a finetuner: if your task touches audio (speech transcription, audio-visual Q&A, spoken captioning), the VL family is the wrong tool — there is no audio path. Use Qwen2.5-Omni-7B for small / dense workloads or Qwen3-Omni-30B-A3B for the MoE tier. For pure-text tasks use Qwen3 / Qwen3.5 dense. For pure-VL-without-audio (this project's case), 2.5-VL / 3-VL / 3.5 are the right targets.

trap · cross-generation processor reuse A processor loaded from a Qwen2.5-VL checkpoint will silently drop Qwen3-VL timestamp tokens and cannot handle Qwen3-Omni audio pads at all. Always call AutoProcessor.from_pretrained(same model_id) as the model. Do not pickle a processor from one generation and load it for another.
trap · OpenAI image_url vs Qwen image OpenAI-style {"type":"image_url","image_url":{"url":"data:..."}} is rejected by process_vision_info. Either convert to Qwen-native {"type":"image","image":<pil|path|url>} before the call, or decode the data-URL yourself and feed PIL images directly to the processor (this is what our project does — see §09).
trap · Qwen3-Omni audio-in-video consistency use_audio_in_video=True must be set the same way for every turn of a multi-round conversation. Flipping it mid-chat breaks TMRoPE time alignment silently (the model still generates, the output is just wrong).
05 finetuning implications of the family tree

What the architecture forces you to change

The naming of linear layers, chat-template tokens, quantisation rules, and which components to freeze are shared across 2-VL / 2.5-VL / 3-VL in broad strokes — but the landmines move. Everything below is the cross-family rule-of-thumb, with per-generation notes where they diverge.

decision Qwen2-VL Qwen2.5-VL Qwen3-VL
LoRA target_modules q_proj k_proj v_proj o_proj gate_proj up_proj down_proj same same (add MoE router caveat)
vision tower freeze default freeze default freeze default; DeepStack indices need explicit skip on 4-bit
chat tokens <|im_start|> <|vision_start|> <|image_pad|> same same + interleaved timestamp tokens (emitted by processor)
QLoRA + vision train unsupportedmust be --bits 16 unsupportedsame rule unsupportedsame rule
flash-attn 2 ok ok ok
dtype bf16 native · fp16 flaky on ViT softmax same same
bnb 4-bit LLM only · skip visual LLM only · skip visual LLM only · skip visual + deepstack_visual_indexes
YaRN + grounding — avoiddistorts M-RoPE temporal handled by Interleaved-MRoPE; fewer caveats

Why the vision tower is frozen by default: PEFT only attaches LoRA to nn.Linear/nn.Embedding leaves; the vision tower modules (attn.qkv, attn.proj, mlp.fc1, mlp.fc2 in 2-VL / 2.5-VL) use different names than the LLM — passing the whole model to get_peft_model with the LLM target list leaves the ViT correctly untouched. If you need vision-side LoRA, opt in explicitly (e.g. --vision_lora in the 2U1/Qwen-VL-Series-Finetune repo) and drop to bf16.

07 hardware · what fits where

VRAM budget — 7B / 8B tier, order-of-magnitude

Numbers shift with max_pixels, seq length, batch size, and gradient-checkpointing. Rule of thumb for a single-GPU researcher: if you can run inference comfortably, you can run QLoRA with frozen ViT.

regime Qwen2-VL-7B Qwen2.5-VL-7B Qwen3-VL-8B

inference 4-bit NF4~6–8 GB~6–8 GB~8–10 GB

Consumer-GPU reality check (RTX 4060, 8 GB): Qwen2.5-VL-7B is not realistic even with QLoRA — the vision encoder alone pushes past 8 GB. Drop to Qwen3-VL-4B or Qwen2.5-VL-3B with load_in_4bit=True, bnb_4bit_quant_type="nf4", paged_adamw_8bit, ctx≤2048, batch=1, grad_accum=8–16, gradient checkpointing on.

Our run: AWS g5.xlarge, 1× A10G (22.06 GiB usable), Ubuntu 24.04 LTS, CUDA 13.0, driver 580. 44 min wall-clock, 1,680 optimizer steps at ~38 steps/min. VRAM headroom ~850 MiB by end of step (tight).

Small-model LoRA recipe · known-good baseline

Across DataCamp's Qwen3-VL-8B tutorial, Unsloth's Qwen3-VL / Qwen3.5 docs, and NVIDIA's DGX-Spark notes, the same configuration recurs. Use it as your starting point — then tune only the pixel budget and batch/accum for your card. 

lora = LoraConfig(
    r=16,                    # small models ≤ 8B · 32 is overkill and OOMs on 12 GB
    lora_alpha=32,           # α = 2·r
    lora_dropout=0.05,
    target_modules=["q_proj","k_proj","v_proj","o_proj",
                    "gate_proj","up_proj","down_proj"],
    task_type="CAUSAL_LM", bias="none",
)

args = TrainingArguments(
    per_device_train_batch_size=2,       # vl-8B: drop to 1 on 24 GB
    gradient_accumulation_steps=4,       # effective batch 8 — stable
    learning_rate=1e-4,                  # LoRA; full-FT drop to 1e-5 / 2e-5
    lr_scheduler_type="cosine",
    warmup_ratio=0.05,
    bf16=True,
    gradient_checkpointing=True,
    optim="adamw_torch_fused",
    num_train_epochs=2,                   # small data? 2–3 epochs · longer = overfits
)

Pixel-budget lever · the one you'll actually use

VRAM on VL training is dominated by the vision token count, not the LLM size. Halving max_pixels typically halves activation memory. Start conservative:

scenariomin_pixelsmax_pixelsmax image sideeffect

hard constraint · Qwen3.5 + bnb 4-bit Unsloth's docs are explicit: QLoRA on Qwen3.5 is unstable because the Gated-DeltaNet hybrid layers don't round-trip cleanly through bnb 4-bit. Use bf16 or 8-bit LoRA. If you're on an 8 GB card and want VL, stay on Qwen3-VL-2B instead.
data-quality filter · brutal DataCamp's Qwen3-VL-8B tutorial retained only 39% of an 84 K-sample dataset after image/aspect-ratio/token-length filtering. Plan on discarding 40–60 % of naively-scraped VL pairs. Running with trash samples wastes more than compute — it pushes the model toward "safe" outputs.
transformers ≥ 4.57 for Qwen3-VL Older transformers need trust_remote_code=True to load Qwen3-VL. Skip that by bumping transformers to >=4.57.0; native support lands there. Same rule for Qwen3-Omni and Qwen3.5.
unsloth + liger-kernel on Qwen3-VL full-FT The combination conflicts on full finetune (LoRA is unaffected). Either disable liger-kernel or switch to DeepSpeed ZeRO-2 for full-FT. Rarely matters for small-model work — LoRA is usually the right tool.
our project, re-run on smaller models The exact same pipeline in notebooks 05–07 works for Qwen3-VL-2B / 4B by swapping BASE_MODEL, transformers>=4.57, and the model class to Qwen3VLForConditionalGeneration. Expect faster epochs (~20 min on A10G for Qwen3-VL-2B vs 44 min for Qwen2.5-VL-7B) and lower CER on long refs — but watch the short-ref fallback mode, it's the same LM-prior leak pattern on smaller models too.

Message format — the silent trap

The JSONL stores OpenAI-style messages. Qwen's qwen_vl_utils.process_vision_info expects a different shape. Get this wrong and you train against empty vision slots (run-01).

our jsonl · OpenAI-style
{
  "messages": [
    {"role": "system", ... },
    {"role": "user",
     "content": [
       {"type": "image_url",
        "image_url": {
          "url": "data:image/png;base64,…"
        }}
     ]},
    {"role": "assistant",
     "content": "{\"transcription\":…}"}
  ]
}
→
qwen-native · what process_vision_info wants
{
  "role": "user",
  "content": [
    {"type": "image",
     "image": <path | PIL | url>},
    {"type": "text",
     "text": "Transcribe"}
  ]
}

We kept the OpenAI shape (portability with existing tools) and wrote a 20-line extract_images() helper that walks the messages, unwraps image_url.url, strips the data:image/png;base64, prefix, base64-decodes through io.BytesIO, and returns a list of PIL RGB images. The processor accepts PIL directly. 

def extract_images(messages):
    """Unwrap OpenAI data-URL images into PIL RGB list."""
    imgs = []
    for m in messages:
        content = m.get("content", [])
        if not isinstance(content, list): continue
        for ele in content:
            if ele.get("type") != "image_url": continue
            url = ele["image_url"]["url"]
            b64 = url.split(",", 1)[-1]
            raw = base64.b64decode(b64)
            imgs.append(Image.open(io.BytesIO(raw)).convert("RGB"))
    return imgs
pixel bounds · must match across notebooks MIN_PIXELS = 4*28*28 = 3136 and MAX_PIXELS = 128*28*28 = 100352. Set identically in the data-transform step (nb 04), the training processor init (nb 05) and the eval processor init (nb 06). Mismatch produces silent token-count drift. Halving MAX_PIXELS is the first OOM lever.
dataset size · one-line file Each row is a self-contained JSON object with the image embedded as a base64 data URL (~17 KB/image). Train file 19.6 MB, eval 5.0 MB. Self-contained → portable; the tradeoff is size, fine at 1,400 samples, painful at 10 M.
10 model loading · 4-bit qlora in 18 lines

BitsAndBytes + PEFT + Qwen2.5-VL

7 B parameters in fp16 ≈ 14 GB. NF4 4-bit compresses that to ~4 GB, freeing ~11 GB for activations, gradients, and optimizer state. bnb_4bit_use_double_quant=True quantises the quantisation scales themselves, saving another ~0.5 GB. 

from transformers import (
    Qwen2_5_VLForConditionalGeneration,
    AutoProcessor, BitsAndBytesConfig,
)
from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training

bnb = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,
)

model = Qwen2_5_VLForConditionalGeneration.from_pretrained(
    "Qwen/Qwen2.5-VL-7B-Instruct",
    quantization_config=bnb,
    torch_dtype=torch.bfloat16,
    device_map="auto",
    attn_implementation="sdpa",            # flash-attn 2 also ok
)
processor = AutoProcessor.from_pretrained(
    "Qwen/Qwen2.5-VL-7B-Instruct",
    min_pixels=3136, max_pixels=100352,

model = prepare_model_for_kbit_training(model)     # casts norms to fp32, enables grad ckpt
lora = LoraConfig(
    r=32, lora_alpha=64, lora_dropout=0.05,
    target_modules=["q_proj","k_proj","v_proj","o_proj",
                    "gate_proj","up_proj","down_proj"],
    task_type="CAUSAL_LM", bias="none",
)
model = get_peft_model(model, lora)            # trainable: ~1.93% / 159 M of 8.29 B

Target modules · anatomy

The LLM backbone of each Qwen transformer block has seven nn.Linear layers worth attaching LoRA to:

  • q_proj · query projection
  • k_proj · key projection
  • v_proj · value projection
  • o_proj · attention output projection
  • gate_proj · SwiGLU gate branch
  • up_proj · SwiGLU up branch
  • down_proj · SwiGLU down projection

Covering all seven is more comprehensive than most defaults (which stop at q_proj/v_proj). The extra coverage costs a few % extra VRAM but delivers materially lower eval loss.

r=32, α=64. Rank 32 is the max stable setting on 24 GB A10G with batch=1 + grad-ckpt for Qwen2.5-VL-7B. Rank 64 OOMs. α = 2×r is the textbook scaling. Dropout 0.05 is a light regulariser appropriate for a 1,120-sample corpus. With bias="none" LoRA skips bias adaptation (keeps parameter count down).

Trainable: ~159 M params · 1.93% of the 8.29 B total. Everything else is frozen and 4-bit quantised.

Dataset — the single highest-ROI fix in the whole project

The run-01 ArabicHTRDataset.__getitem__ called self.processor(text=text, return_tensors="pt") without passing images=. The model trained against empty vision slots for 1 h; at eval it hallucinated a 512-token Arabic loop. The fix is four lines: 

class ArabicHTRDataset(Dataset):
    def __getitem__(self, idx):
        sample   = self.samples[idx]
        messages = sample["messages"]
        text     = self.processor.apply_chat_template(
            messages, tokenize=False, add_generation_prompt=False,
        )
        image_inputs = extract_images(messages)      # ← THE FIX

        inputs = self.processor(
            text=[text],
            images=image_inputs or None,          # ← THE FIX
            return_tensors="pt",
            truncation=True,
            max_length=8192,
            padding=False,                             # ← dynamic pad via collator
        )
        inputs["labels"] = inputs["input_ids"].clone()
        return {k: v.squeeze(0) for k, v in inputs.items()}

Pair with DataCollatorForSeq2Seq(tokenizer=processor.tokenizer) so batches pad to the longest sample and -100 fills label positions at pad slots — standard causal-LM masking.

OOM trap · padding="max_length" Hard-coding padding="max_length" at MAX_SEQ_LENGTH=8192 pads every sample — even 12-token ones — to 8 K. Activations are quadratic in seq length, and a single short sample blew 4.64 GiB. Switch to dynamic padding via DataCollatorForSeq2Seq and the problem disappears. Secondary levers (in order of impact): drop LORA_R 32 → 16, halve MAX_PIXELS, expandable_segments:True.
loss trajectory · run-02 train 8.01 → 1.46 · eval by epoch 1.6350 / 1.6248★ / 1.6387 · best = ckpt-1120 (end of epoch 2). Gradient-norm drifted from ~2 to ~4 through training; clipping at 1.0 kept things stable. Epoch 3 is a textbook mild-overfit signature: train keeps falling, eval turns up. Early-stop at epoch 2 in future runs.

Drivers: training ref-length distribution biases output length, the JSON prefix primes continuation not termination, and repetition_penalty doesn't reward EOS.

13 inference · vllm + lora swap

Serve the adapter without merging

vLLM supports hot-loading LoRA adapters on top of a base model, so you don't need to materialise a merged 14 GB checkpoint. The --max-lora-rank flag must match the adapter's r. 

$ python3 -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-VL-7B-Instruct \
    --enable-lora \
    --max-lora-rank 32                          # matches LoRA r \
    --lora-modules arabic-ocr=/path/models/lora_adapter/run-2 \
    --served-model-name arabic-ocr-v1 \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.90 \
    --trust-remote-code \
    --host 0.0.0.0 --port 8000 \
    --dtype auto

OpenAI-compatible client — drop-in replacement for the ChatCompletions SDK, same request shape as your data-labelling code: 

from openai import OpenAI
client = OpenAI(base_url="http://localhost:8000/v1", api_key="sk-any")
resp = client.chat.completions.create(
    model="arabic-ocr-v1",
    messages=[{
        "role": "user",
        "content": [
            {"type": "image_url",
             "image_url": {"url": f"data:image/png;base64,{b64}"}},
            {"type": "text", "text": "Transcribe."},
        ],
    }],
    max_tokens=128, temperature=0,
)

vLLM uses PagedAttention and speculative decoding — expect 3–10× throughput over HF model.generate on the same hardware. For 24 GB A10G, leave --gpu-memory-utilization ≤ 0.90 and keep --max-model-len aligned with your training setting (8 K for this project) so KV-cache doesn't balloon.

14 issues encyclopedia · every scar

Fourteen concrete failures hit during runs 01–02, grouped by where they surface. Every card = exact error + root cause + the one-line fix.

deps · import
ImportError: cannot import Qwen2_5_VLForConditionalGeneration
Cause: pinned transformers==4.47.0; class added in ≥ 4.52. fix pip install "transformers>=4.52.0" then restart kernel.
deps · bnb
Using bnb 4-bit requires bitsandbytes>=0.46.1
Cause: bitsandbytes==0.45.0 pre-dates the 4-bit hook in newer transformers. fix pip install -U "bitsandbytes>=0.46.1" · we landed on 0.49.2.
deps · accelerate
TypeError: Accelerator.unwrap_model() got unexpected kwarg 'keep_torch_compile'
Cause: transformers ≥ 4.52 calls the kwarg, accelerate==1.2.1 doesn't know it. fix pip install "accelerate>=1.3.0". Keep transformers + accelerate in lockstep.
deps · processor
AutoProcessor.from_pretrained() crashes on Qwen2.5-VL
Cause: the processor's internal Qwen2VLVideoProcessor imports torchvision, even for image-only workflows. fix pip install torchvision. Not listed in model card.
deps · linter
prepare_model_for_kbit_training is not exported from peft
Cause: linter uses type stubs that moved the symbol to peft.utils.other. Runtime import from top-level still works. fix Silence linter with from peft.utils.other import prepare_model_for_kbit_training · functionally identical.
deps · rename
evaluation_strategy is a no-op in transformers ≥ 4.46
Cause: argument renamed to eval_strategy. Only surfaces as a linter warning; silently ignored. fix eval_strategy="epoch".
memory · oom
OutOfMemoryError: CUDA out of memory · tried to allocate 4.64 GiB
Cause: padding='max_length' + MAX_SEQ_LENGTH=8192 pads every sample to 8 K tokens. Activations are O(n²) in seq length. fix padding=False in the processor call, wire DataCollatorForSeq2Seq into the Trainer. Then optionally LORA_R 32→16, halve MAX_PIXELS, and PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True.
vision · silent
empty pixel_values — model trains on blank vision slots
Cause: Dataset called processor(text=...) without images=. The model got no visual input. Eval exploded into runaway Arabic. fix Build extract_images() helper, pass images=image_inputs or None to the processor. Single highest-ROI bug in the project.
vision · format
ValueError: image, image_url or video should in content
Cause: qwen_vl_utils.process_vision_info expects Qwen-native {"type":"image","image":...}, not OpenAI {"type":"image_url","image_url":{"url":"data:..."}}. It reads ele["image_url"] as a string, finds a dict, throws. fix 20-line extract_images() bypass — see §05.
eval · loop
48 s/sample · model outputs 512-token Arabic monologue
Cause: (1) use_cache=False leaked from training (grad-ckpt forces it off), (2) max_new_tokens=512 too high, (3) no stop_strings, (4) empty vision. fix model.config.use_cache = True + use_cache=True in .generate() + max_new_tokens=128 + stop_strings=['"}'] + torch.inference_mode(). Latency 48 → 3.5 s.
metrics · wrong
CER > 1.0 — looks broken, is actually correct
Cause: jiwer.cer = (S+D+I) / N_ref. When prediction is longer than reference, insertions push the numerator above N. This is the standard definition. Notebook markdown incorrectly claimed "clamped to [0,1]". fix Keep CER raw so over-production stays visible. Fix the root cause with stop_strings and a tighter max_new_tokens.
cache · hf
"Re-downloading 15 GB on every kernel restart"
Cause: not a re-download — from_pretrained does ETag HTTP checks, then loads + NF4-quantises 729 safetensors shards from disk to VRAM (~2 min). fix snapshot_download(BASE_MODEL, local_files_only=True) + from_pretrained(snapshot_path, local_files_only=True). Progress bar now visibly "disk → GPU", not network.
ux · state
NameError: name 'CHECKPOINT' is not defined
Cause: after a kernel restart, the config cell was skipped; the summary cell later referenced CHECKPOINT. fix globals().get('CHECKPOINT', None) or 'merged' and rebuild LOG_DIR from RUN_NAME defensively.
os · pep668
error: externally-managed-environment
Cause: Ubuntu 24.04's system Python blocks pip install globally (PEP 668). fix Create a venv at .venv; notebook install cells use !{sys.executable} -m pip install ... so they target the kernel interpreter, not /usr/bin/python3.

16 the pre-flight checklist · if i did this again
✓
Verify pixel_values is non-zero before step 1. Assert batch["pixel_values"].abs().sum() > 0 in a training hook. The single most expensive bug in this project was silent vision absence.
✓
Pin min-versions, not exact versions. transformers≥4.52 · accelerate≥1.3 · bitsandbytes≥0.46.1 · peft==0.14.0 · torchvision · qwen-vl-utils · jiwer.
✓
Dynamic padding from day one. DataCollatorForSeq2Seq · never padding='max_length' on an 8 K context.
✓
Use the model's own processor. Qwen2-VL ≠ Qwen2.5-VL ≠ Qwen3-VL processor signatures (return_video_kwargs, timestamp tokens).
✓
Freeze the vision tower unless you have a concrete reason to train it. If you do, drop 4-bit → bf16 (QLoRA + vision train is unsupported).
✓
Restart the kernel after every pip install -U. Python caches module objects; stale bytecode produces misleading errors.
✓
Re-enable use_cache at eval. Gradient checkpointing turns it off; explicit model.config.use_cache = True + pass use_cache=True to .generate().
✓
Evaluate every checkpoint. Don't auto-ship the final. Trust trainer_state.json's best_global_step.
✓
Stratify CER by reference length before reporting. A single mean hides over-production in the short bucket vs. a sane long-ref regime.
✓
Use jiwer for CER/WER. Don't clamp — values > 1 are meaningful (over-production). Fix the cause, don't hide the signal.
✓
Force JSON prefix at decode time ({"transcription":") and pair with stop_strings=['"}']. Gets you a 100% valid-JSON rate.
✓
Snapshot-download the base model. snapshot_download(local_files_only=True) then from_pretrained(snapshot_path, local_files_only=True). Eliminates "re-download" illusions on kernel restart.
✓
Match --max-lora-rank to the adapter's r when launching vLLM. Forgetting this is a common rejection reason.
✓
Expect epoch 2 to be the keeper on small datasets. 1 k samples → 3 epochs is the minimum floor; epoch 3 usually overfits. Add weight decay if you must run further.
