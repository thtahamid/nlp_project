01 why this project exists

The problem, in one picture

Arabic handwriting recognition is hard. Scripts are cursive, letters change shape based on position, diacritics are optional, and there is no standardized script form across writers. Classical OCR engines (Tesseract, EasyOCR) were built for printed, Latin-script pages — they fall apart on handwriting.

The modern answer is a Vision-Language Model (VLM) — one model that reads an image and emits text, trained the same way a chat model is. These models already know Arabic script from pretraining; fine-tuning teaches them your specific handwriting style, layout, and output format.

What "a project like this" means

  • Input: an image containing handwriting (a scanned page, a crop, a photo).
  • Output: structured text — in this repo's case, a JSON payload like {"transcription":"…"}.
  • Method: take a pretrained 7B VLM (Qwen2.5-VL-7B), add a small trainable adapter (LoRA), train on ~1,400 image/label pairs, evaluate on a held-out 20%, serve with vLLM.
  • Constraint: all of this has to fit on a single 24 GB consumer GPU (e.g. RTX 3090, A10G).
Who this guide is for You have written some Python, you know what a neural network is in general terms, and you want to understand every layer of the stack — from "what is a token" up to "why does vLLM's PagedAttention make inference 10× faster." By the end you should be able to open any script in this repo and read it like prose.
02 nlp fundamentals · tokens & language models

A language model is a probability function over tokens

Forget the magic for a moment. At its core, a language model is a function that answers one question over and over again:

Given the tokens I've seen so far, what is the probability distribution over the next token?
Formally: P(tn | t1, t2, …, tn-1)

Everything else — text generation, chat, summarization, transcription — is just calling that function repeatedly and feeding each predicted token back in. This is called autoregressive or causal language modeling, and it's the paradigm behind GPT, Llama, Qwen, Claude, and every model in this project.

What is a "token", really?

Models don't see characters or words. They see token IDs — integers into a fixed-size vocabulary (~150k entries for Qwen). A tokenizer is the deterministic function that maps raw bytes/text to that ID sequence and back. Modern tokenizers use Byte-Pair Encoding (BPE): start with single bytes, then iteratively merge the most common adjacent pairs until you hit the target vocabulary size.

chars 0
tokens 0
chars/token 0
vocab size ≈ 152k
why it matters Tokens are the unit of everything — price per 1M tokens, the max_seq_length=8192 in config.py, the max_new_tokens=512 at inference, VRAM usage during training. Arabic typically costs 2–3× more tokens per character than English because of how BPE handles non-Latin scripts; that's why the model's context window matters so much for our use case.

Embeddings: from integers to vectors

A token ID is just an integer. To do math on it, the model looks it up in an embedding table — a giant matrix of shape [vocab_size × hidden_dim]. For Qwen2.5-7B that's roughly 152,064 × 3,584 floats — about 545 million parameters in the embedding layer alone.

Each row is a learned point in a high-dimensional space. Tokens with related meanings cluster together; the geometry of that space encodes everything the model knows about language. Fine-tuning rarely touches these directly — the embeddings are frozen and we adjust only the transformer layers that read from them.

Vocabulary

03 the transformer · attention is all of it

Every modern LLM is a stack of identical transformer blocks

The transformer is an architecture published in 2017 ("Attention is All You Need") that replaced recurrent networks. A model like Qwen2.5-7B is just 28 identical blocks stacked on top of each other, each doing the same operation: let every token look at every other token, combine what they say, and update its own representation.

Self-attention in 40 seconds

Inside each block, every token projects itself into three vectors: a query ("what am I looking for?"), a key ("what do I offer?"), and a value ("if someone attends to me, here's what they get"). Every token compares its query against every other token's key (dot product → softmax → weights), then gathers a weighted sum of their values. The output is a token's new, context-aware representation.

attention(Q,K,V) = softmax( Q·KT / √dk ) · V
Hover a query token to see where attention flows. Darker cells = higher weight.
In causal attention, each token can only see itself and earlier tokens — never the future. That's why generation happens one token at a time.

Inside one transformer block

  1. LayerNorm — normalize the input so the gradients behave.
  2. Multi-head attention — run self-attention in parallel across multiple "heads" (each head learns a different kind of relationship) and concatenate.
  3. Residual add — x + attention(norm(x)).
  4. LayerNorm — normalize again.
  5. Feed-forward network (FFN) — two linear layers with a nonlinearity in between. This is where most parameters live: gate_proj, up_proj, down_proj in Qwen (SwiGLU variant).
  6. Residual add — x + ffn(norm(x)).
why residuals matter Every block adds to the signal instead of replacing it. This lets gradients flow backward through the whole 28-block stack without vanishing, and it's why you can fine-tune with a small delta (LoRA) without rewriting the whole model — you're just adding a correction to each residual stream.

The projection layers fine-tuners care about

q_proj · k_proj · v_proj
attention projections
Map the residual stream → query / key / value vectors. First choice for LoRA targets — changing them changes how the model routes attention.
o_proj
attention output
Combines the outputs of all attention heads back into the residual stream.
gate_proj · up_proj
FFN input side
Two parallel linear layers (SwiGLU) that expand the hidden dimension by ~4×.
down_proj
FFN output side
Compresses the FFN's wide hidden state back to the residual dimension. Most parameters per layer live here.

In this repo's config.py, LORA_TARGET_MODULES is set to all seven: q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj. That's deliberate — covering both attention and the FFN gives the adapter enough leverage to change both what the model attends to and what it does with that information.

04 vision-language models · images as tokens

A VLM is an LLM with one extra move: images also become tokens

The decoder side of a Vision-Language Model is just a regular LLM. The magic is in the vision encoder — a separate network (usually a Vision Transformer, or ViT) that takes an image and produces a sequence of vectors that look like token embeddings to the decoder. The LLM can't tell images and text apart at that point; everything is just vectors in a shared space.

From pixels to patches to tokens

raw image
448 × 448 px
→
visual tokens
~256 tokens
  1. The image is resized to a size that fits the pixel budget — in this repo: min_pixels=3,136, max_pixels=100,352. Qwen2.5-VL supports dynamic resolution, so bigger images = more tokens.
  2. It's cut into non-overlapping patches (typically 14×14 or 28×28 px).
  3. Each patch is flattened and run through a linear "patch embedding" layer → a vector of size hidden_dim.
  4. Position encodings are added so the model knows where each patch came from in the image.
  5. A few transformer blocks process the sequence, then a projection layer (called a "connector" or "merger") maps the visual sequence into the same vector space the LLM uses for its text tokens.
the key insight After projection, the vision sequence is indistinguishable from a text-token embedding sequence to the decoder. So we can interleave them: [text_tokens ··· image_tokens ··· more_text_tokens]. This is what makes "Transcribe this image: [IMG] → output" work as a single forward pass.

Chat templates — how images end up in a prompt

Every chat model has a chat template, a Jinja-style string that turns a list of messages into the exact flat token sequence the model was trained on. For Qwen2.5-VL it looks (conceptually) like this: 

<|im_start|>system
You are a transcription assistant.<|im_end|>
<|im_start|>user
<|vision_start|>[IMAGE_TOKENS]<|vision_end|>Transcribe this.<|im_end|>
<|im_start|>assistant
{"transcription":"…"}<|im_end|>

The AutoProcessor from 🤗 Transformers handles all of this: it resolves the image, runs the vision encoder, formats the tokens with the special markers, and returns tensors ready for the model. In this repo you'll see it built in train.py and evaluate.py.

05 pretrain → fine-tune · the two-stage playbook

You don't train a foundation model. Someone else did. You fine-tune.

The cost of pretraining Qwen2.5-7B from scratch is on the order of millions of dollars and months of compute. What makes projects like this one possible is that Alibaba shipped the weights publicly — we get the benefits of that investment, then teach the last 1% with our own tiny dataset.

Stage 1 · Pretraining
done by others · months · $M
Train on trillions of tokens of web text, books, code, and (for VLMs) image-caption pairs. Objective: next-token prediction on everything. Result: a model that understands language and vision, but has no opinion about your specific task.
Stage 1.5 · Instruction tune
done by vendor · weeks
A second pretraining phase on (instruction, response) pairs to teach the model to follow instructions in a chat format. The "-Instruct" suffix in Qwen2.5-VL-7B-Instruct signals this step was done.
Stage 2 · Fine-tune (YOU)
hours · $10–$50 on cloud
Train on your own dataset (in this repo: 1,120 Arabic HTR pairs). Objective: next-token prediction on your task format. Only a small adapter changes — the rest of the model is frozen.

Knowledge distillation — where the labels come from

Projects like this one often use a strong model (GPT-5.4-mini in this repo) to generate reference labels for the training set, then fine-tune a smaller/cheaper model to reproduce them. This is knowledge distillation — you're transferring capability from the teacher model into your student model. The student never outperforms the teacher on the exact task, but it's dramatically faster, cheaper, and can be self-hosted.

what "teaching" actually changes Fine-tuning doesn't give the model new facts. It biases the next-token distribution toward outputs that look like your training data. If your training set is 1,400 JSON transcriptions, the model learns "when I see a user asking to transcribe an image, answer with JSON." Everything else it already knew (Arabic script, JSON syntax) came from pretraining.
06 data engineering · what the model actually eats

Garbage in, garbage out — with extra steps

Fine-tuning quality is bounded by dataset quality. In this repo the pipeline has two data stages: collect (raw images → reference labels) and transform (raw labels → model-ready chat JSONL).

The JSONL discipline

Every dataset in this repo is a JSONL file: one JSON object per line, newline-delimited. It's the universal format for ML datasets because it's streamable (no need to load the whole file), diff-able (line-by-line history), and incremental (write as you go, safe against crashes).

data/raw/*.png
stage 0 input
The image assets on disk. Each file is one training example once labeled.
data/gpt_labels.jsonl
after collect
One line per labeled image: {"image": "page_0423.jpg", "transcription": "…"}. Labels come from GPT-5.4-mini via the OpenAI API.
data/train/train.jsonl
after transform
Chat-format samples. Image is embedded as a data:image/png;base64,… URL — the entire dataset is self-contained.
data/eval/eval.jsonl
held out
20% of samples, never touched by training. This is the only honest signal of generalization.

Image preprocessing: the pixel budget

Qwen2.5-VL accepts variable resolution, but converting bigger images means more visual tokens, which costs VRAM at training time and latency at inference time. This repo uses a pixel budget to clamp both ends:

  • MIN_PIXELS = 3,136 — roughly 56×56 px. Anything smaller is upscaled with LANCZOS (a high-quality resampler) so the vision encoder has enough signal.
  • MAX_PIXELS = 100,352 — roughly 317×317 px. Anything larger is downsampled. At this budget each image consumes ~256 visual tokens — a known cost.
train/eval split hygiene The split is random with a fixed seed (42) and never re-drawn. If you reshuffle between experiments you're measuring noise, not progress. Eval samples never, ever appear in training — that's the contract that makes eval loss and CER meaningful.

Chat-format schema

Each line in train.jsonl is a conversation with a system instruction, a user turn that includes the image, and an assistant turn that contains the target output. The trainer masks out the system and user tokens from the loss — the model is only penalized for what it emits as the assistant.

07 lora & peft · cheap adaptation

Don't fine-tune 8 billion parameters. Fine-tune a handful.

Parameter-Efficient Fine-Tuning (PEFT) is the umbrella term for techniques that train only a small fraction of the model. The most popular by far is LoRA — Low-Rank Adaptation.

The math in one line

For every weight matrix W ∈ ℝd_out × d_in you want to adapt, LoRA freezes W and learns two small matrices A ∈ ℝr × d_in and B ∈ ℝd_out × r, with rank r ≪ min(d_in, d_out). At inference the effective weight is:

W' = W + (α/r) · B·A
During the forward pass, the model computes both W·x and B·(A·x) and adds them. Only A and B receive gradients.

The four dials you actually touch

r (rank)
capacity of the adapter
Higher r → more expressive adapter, more params, more VRAM, slower. Typical: 8, 16, 32, 64. This repo: r=32.
alpha
scaling factor
The update is (α/r)·BA, so alpha controls how strongly the adapter contributes. Convention: α = 2·r. This repo: α=64.
dropout
regularization
Applied on the LoRA path (not the base weights). Prevents the adapter from memorizing the training set. Typical: 0.05–0.1.
target_modules
where to attach
Which layer names get LoRA. Minimum: q_proj, v_proj. For best results: all 7 attention + FFN projections.
why LoRA works at all Empirically, task-specific fine-tuning lives on a low-dimensional manifold relative to the full parameter space. You don't need to wiggle 8 billion parameters — you need to wiggle a few million in the right direction. LoRA's rank-r decomposition restricts updates to exactly that low-dimensional subspace, which also acts as implicit regularization.

Related PEFT techniques

  • QLoRA — LoRA on top of a 4-bit quantized base. This repo uses it. The base model is loaded in NF4 and frozen; the LoRA matrices are in bf16 and trained.
  • DoRA (Weight-Decomposed LoRA) — decomposes weights into magnitude + direction and adapts them separately. ~5-10% better than LoRA on some tasks.
  • Prefix tuning / prompt tuning — learn a small number of "virtual" prompt tokens instead of touching weights. Cheaper than LoRA but less expressive.
  • IA³ — learn a per-channel scaling vector. Even fewer params than LoRA, worse expressivity.
08 quantization · lower precision, less vram

An 8B model in FP32 needs 32 GB of VRAM. In NF4, it needs 4 GB.

A model's weights are just floating-point numbers. The question is: how many bits per number? Fewer bits = less VRAM, less bandwidth, faster compute — at the cost of precision. Modern fine-tuning lives on a careful balance between these extremes.

The five precisions that matter

FP32
standard float
1 sign + 8 exp + 23 mantissa bits. Baseline for correctness but 2–8× more memory than anyone wants.
FP16
half precision
1 + 5 + 10 bits. Half the memory, but the exponent range is small — gradients can underflow. Needs loss scaling.
BF16
"brain float"
1 + 8 + 7 bits. Same exponent range as FP32, worse mantissa. Robust for training, first choice on Ampere/Ada/Hopper GPUs.
INT8
8-bit integer
8 bits per weight with a per-channel scale factor. Lossy but usually acceptable for inference.
NF4
4-bit NormalFloat
4 bits per weight, quantization levels chosen to match a normal distribution (which pretrained weights follow). The star of QLoRA.
double quantization To compress NF4 further, the quantization constants themselves (the per-block scale factors) are quantized a second time. Saves another ~0.4 bits per parameter. That's what bnb_4bit_use_double_quant=True does in this repo's config.

The mixed-precision recipe you'll see in this repo

  • Base model weights — stored in NF4 (4 bits), frozen.
  • Compute dtype — bfloat16. When doing a matmul, the 4-bit weight is dequantized to bf16, the math happens in bf16, the result is bf16.
  • LoRA adapter weights — bfloat16 (in FP trainable).
  • Gradients — accumulated in float32 for numerical stability.
  • Optimizer state — first/second moments (AdamW) in float32.
09 memory budget · where the 24 gb goes

Training a 7B model on 24 GB is a memory puzzle

At a rough estimate, naive full fine-tuning of a 7B model in FP32 needs ~112 GB of VRAM — 28 GB for weights, 28 for gradients, 56 for the AdamW optimizer state. We don't have 112 GB. We have 24.

Gradient checkpointing — trading 25% speed for 50% memory

During a forward pass, the model saves every intermediate activation so it can use them in backprop. For a 28-layer 7B model, activations are the single largest memory consumer. Gradient checkpointing tells the trainer: "forget most of those. When you need them for backprop, recompute them." You pay with an extra forward pass per step (~25% slowdown) and save ~50% of activation memory. This is the difference between OOM and working on a 24 GB card.

why grad-checkpointing fights with use_cache At inference, use_cache=True stores the KV cache, which is another form of cached activations. During training, gradient checkpointing sets use_cache=False automatically — trying to cache things it's going to throw away would be silly. You have to set model.config.use_cache = True explicitly before running eval or inference.

Cross-entropy loss in plain words

After the forward pass, for every target token position, the model produces a vector of logits over the whole vocabulary. We apply softmax to get a probability distribution, then compute:

loss = −log( P(correct_token) ) (averaged over all target positions)

If the model assigned probability 1.0 to the correct next token, loss is 0. If it assigned probability 0.001, loss is ~6.9. This scalar is what backpropagation chews on.

The AdamW optimizer

Plain SGD says "step in the opposite direction of the gradient." AdamW remembers a running average of past gradients (first moment) and their squared magnitudes (second moment), then takes a step that's scaled per-parameter by how confident we are. Practically: training is more stable, converges faster, is less sensitive to learning rate. Every fine-tune in this repo uses AdamW.

The learning rate & its schedule

lr=2e-5 in this repo — meaning at each step, AdamW's preconditioned gradient is multiplied by 0.00002 before being applied. Too high and you lurch past the optimum; too low and you never get there. The scheduler adjusts this over time:

  • Warmup (warmup_ratio=0.1) — first 10% of steps, LR ramps linearly from 0 up to 2e-5. Prevents early instability when gradients are huge.
  • Cosine decay — after warmup, LR follows a half-cosine curve down toward 0. Smoother than linear, empirically works best.

Effective batch, revisited

The training glossary covers this in depth, but briefly: one step is one call to optimizer.step(). You can split that step across multiple micro-batches (batch_size samples at a time) and accumulate their gradients (grad_accum_steps times) before applying. Your effective batch is batch_size × grad_accum. This repo: 1 × 2 = 2.

11 inference · generating one token at a time

Training is a single pass. Inference is hundreds of passes.

At inference, you feed the prompt forward once to get the next token, append it, feed the whole thing forward again, get the next token, append, repeat. That's autoregressive decoding. Every token is one full model evaluation, so long outputs are expensive.

Transcribe this image: [IMG] →

Decoding strategies — how we pick the next token

Greedy
argmax
Pick the single most likely token every time. Deterministic. This repo's choice at eval: do_sample=False.
Temperature sampling
soft random
Divide logits by T, softmax, sample. T=0.1 is near-greedy; T=1.0 is the raw distribution.
Top-k / top-p
tail-cutting
Only consider the top-k most likely tokens, or the smallest set whose cumulative probability exceeds p. Kills unlikely tails.
Beam search
best of N paths
Explore N most likely sequences in parallel, pick the best overall. Uncommon in modern LLMs — slow, can over-optimize for length.

The KV cache — the reason generation isn't O(N²)

Here's the trick. When generating token N+1, the attention computation for all earlier tokens has not changed — the queries, keys, and values for positions 1..N are identical to what they were one step ago. So we cache them. Only the new token does any real work.

Without KV cache
O(N²) · recomputes everything
compute operations: 0
With KV cache
O(N) · reuses prior keys/values
compute operations: 0
the measurable payoff In this repo's Run 02 evaluation, enabling use_cache=True at eval time gave an 11× speedup versus Run 01's default — from ~4 s/sample to ~0.4 s/sample for the generation portion. It's the single biggest inference-side optimization you can make with one line of code.

Structured output — keeping JSON valid

The model is probabilistic, so "always respond with valid JSON" is a suggestion, not a guarantee. This repo uses three cheap tricks to get to 100% valid-JSON rate:

  • Prompt engineering — the system message says "respond with {"transcription":"…"}" and includes examples.
  • JSON-prefix forcing — append {"transcription":" to the prompt before generation so the model only has to complete the value and close the braces.
  • stop_strings — halt generate() the moment "} appears in the output. Stops trailing hallucinations cold and caps wall-clock.

For stricter guarantees, libraries like outlines, jsonformer, guidance, and xgrammar constrain the decoder's token distribution to only permit tokens that make the output match a grammar (JSON schema, regex, context-free grammar). Slight inference overhead, 100% valid output.

12 serving · from notebook to api

Training gives you a folder. Serving turns it into a URL.

After training, models/lora_adapter/ contains ~380 MB of LoRA weights and a config file. That's a checkpoint, not an API. To serve it you need an inference server that:

  • loads the base model + adapter on startup,
  • accepts HTTP requests (ideally in the OpenAI API shape, so any client works),
  • batches incoming requests efficiently across the GPU,
  • manages the KV cache without running out of memory.

Why vLLM instead of plain transformers.generate()

PagedAttention
vLLM's core invention
KV cache is stored in fixed-size "pages" like OS virtual memory, not contiguous blocks. ~2–4× throughput by eliminating fragmentation.
Continuous batching
dynamic scheduling
New requests can join an in-flight batch between decoding steps. No waiting for the slowest request to finish.
LoRA hot-swap
--enable-lora
Multiple LoRA adapters loaded at once; the client picks one per request via the model field. Zero restart cost.
OpenAI API-compatible
drop-in
Speaks /v1/chat/completions out of the box. Any OpenAI SDK, LangChain client, or curl command works unchanged.

Launch flags you must get right

  --trust-remote-code \

  • --max-model-len 8192 — matches MAX_SEQ_LENGTH in config.py. Larger → more KV cache pressure.
  • --served-model-name arabic-ocr-v1 — what clients see in /v1/models. Nice for multi-adapter setups.

The three endpoints you test first

  1. GET /health — returns 200 OK when the server is ready. No GPU work.
  2. GET /v1/models — lists loaded models. Should include arabic-ocr-v1.
  3. POST /v1/chat/completions — the real thing. Same payload shape as OpenAI's API, image is sent as data:image/png;base64,….
13 evaluation · measuring what users care about

Eval loss proves it's learning. Task metrics prove it's useful.

There are two kinds of "evaluation" in this repo and you have to keep them separate.

  • Eval loss — cross-entropy on the held-out JSONL, reported once per epoch during training. Cheap, runs in the training loop. Tells you "is the model still generalizing?" Nothing more.
  • Task evaluation — actually generating transcriptions, parsing them, and computing CER/WER against references. Slow (seconds per sample × 280 samples), happens after training. Tells you "is the model actually good at the job?"

CER and WER from first principles

Both measure edit distance between the prediction and the reference:

(Substitutions + Deletions + Insertions) / Nreference
  • CER — character error rate. Counts edits in characters.
  • WER — word error rate. Counts edits in words.
  • Lower is better, 0 = perfect.
  • Can exceed 1.0 — if the hypothesis is much longer than the reference, insertions outnumber reference tokens. This is the classic gotcha.
the failure mode this repo found Run 02 reported mean CER 1.28, which sounds like catastrophe but wasn't. Stratified by reference length: long refs (>60 chars) sat at a sane 0.83. The headline was dominated by 8 short references (<10 chars) that averaged ~9.0 CER because the model emitted its pretraining prior — a 17-word Arabic sentence — instead of a short answer. Same model, three completely different quality regimes. Always stratify.

Other metrics worth computing

Exact match
strictest
1 if prediction == reference char-for-char. Useful when there's one correct answer.

% of outputs that parse as valid JSON with the expected schema. Sanity check before CER even runs.

Ref-length vs. prediction-length distribution. Spikes reveal length bias and LM-prior fallback.

Best checkpoint vs final checkpoint

Always evaluate at least two checkpoints: the one with lowest eval loss (typically mid-training) and the final one. Compare their task metrics. The final checkpoint is often worse than the mid-training one if overfit started — that's Run 02's story exactly.

The underlying hardware contract

  • NVIDIA GPU with compute capability ≥ 8.0 (Ampere or newer) for bf16. A10G, 3090, 4090, A100, H100 all work.
  • 24 GB VRAM minimum for 7B + LoRA + bf16 activations. A 16 GB card works if you cap max_pixels aggressively.
  • CUDA 12.x and a matching NVIDIA driver.
  • Linux is easiest. Windows works via WSL2; macOS is mostly for development and doesn't run bitsandbytes natively.

16 pitfalls & next steps

The thirteen things that will bite you

OOM on first step
CUDA out of memory
Lower max_pixels, enable gradient checkpointing, drop batch_size to 1, raise grad_accum.
Rank mismatch at vLLM launch
ValueError
Default --max-lora-rank=16 rejects a rank-32 adapter. Set to the rank you trained with.
use_cache still off at eval
slow eval
Grad checkpointing disabled it. Set model.config.use_cache = True before calling generate.
LM prior hallucinations
canned pretraining phrase
When vision signal is weak, the decoder emits a generic sentence. Balance short/long refs; use stop_strings.
CER > 1.0 headlines
stratification needed
A few short-ref failures can dominate the mean. Report stratified CER and/or clip at 1.0.
Final checkpoint ≠ best
overfit
Always evaluate both the min-eval-loss checkpoint and the final one. Ship whichever wins on task metrics.
Tokenizer mismatch
silent disaster
If inference uses a different tokenizer than training, outputs are gibberish. Always load the processor/tokenizer from the same base model ID.
Images too big at load
1 image ≈ 10k tokens
Without max_pixels, a 4K image blows past the 8192 context window. Pin max_pixels explicitly.
Non-deterministic eval
hard to compare runs
Set seed, use do_sample=False, match processor version. Tiny drift is normal; big swings mean a bug.
Dataset leakage
eval contaminated
If the same image landed in train and eval (duplicate filenames, fuzzy matches), eval numbers lie. Hash images, check for dupes.
Forgot to save processor
reload fails
LoRA checkpoints save adapter weights only. Save the processor separately or reload from the base model at inference.
Wrong LoRA target_modules
loss plateaus high
Targeting only q_proj, v_proj misses the FFN. Include gate_proj, up_proj, down_proj for best results.
Merge-then-reload drift
merged ≠ adapter
Merging LoRA into quantized base and saving in fp16 can drift vs. serving adapter directly. Evaluate both.

Where to go from here

  • Balance the dataset — reference-length distribution drives the dominant failure mode. Audit and augment short-ref samples.
  • Try higher LoRA rank — jumping from r=32 to r=64 doubles adapter capacity for minimal cost.
  • Try DoRA — drop-in replacement for LoRA in PEFT that often gives +5-10% on vision tasks.
  • Add grammar-constrained decoding — outlines or xgrammar guarantees valid JSON without needing stop_strings.
  • Read the papers — "Attention is All You Need" (2017), "LoRA" (2021), "QLoRA" (2023), "vLLM / PagedAttention" (2023), "Qwen2.5-VL Technical Report" (2024).
  • Iterate fast — the tightest loop in this repo is config.py → train.py → evaluate.py. Keep experiments under 2 hours and you'll try 10× more ideas.
